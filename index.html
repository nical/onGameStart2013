Under the hood

  • On mobile: OpenGL ES2
  • On Mac and Linux: "desktop" OpenGL
  • On Windows: ANGLE → Direct3D

The content process

  • WebGL runs in the same thread as layout and JS
  • It can be slowed down by the rest even if independent
  • WebGL workers \o/

Security

  • WebGL exposes the user's hardware to web developers.
  • Security is very important on the web
  • The browser needs to perform additional checks
    • Such as...
    • and...
  • Browsers continue to improve

WebGL API

  • The bottleneck is often the communication with the GPU
  • Keep the amount of draw calls low!
  • More generally, keep the amount of API calls low
    • Cache getter calls such as getParameter() and getUniformLocation()

Sharders

  • Higly parallel execution
  • We can render worlds in two triangles!

Branching 

            if (foo < 0.5) {
              // do A...
            } else {
              // do B...
            }
  • Branching is slow!
  • If neighbooring fragments don't follow the same path, often costs like computing both A and B

Sampling textures

  • Reading from a texture in a deterministic way is OK
  • Cache misses are very expensive

Beware shader compile time

  • Shader compilation is synchronous
  • Very heavy shaders can block the browser for several seconds depending on the quality of the drivers

Various tips

  • Avoid high precision floats (mobile hardware don't support it well)
  • Always have vertex attrib 0 array enabled.
    • Otherwise it forces the browser to do complicated emulation when running on desktop OpenGL.
source: http://www.khronos.org/webgl/wiki/WebGL_and_OpenGL_Differences
